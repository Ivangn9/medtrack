1. Introducción

La aplicación CIMA Hub gestiona actualmente el inventario, estados operativos e historial de mantenimiento de los equipos biomédicos del centro. Una extensión natural del sistema es la incorporación de datos ambientales en tiempo real: temperatura, humedad relativa y presión del aire de las salas donde opera el equipamiento crítico.

Este documento describe la arquitectura técnica recomendada para integrar sensores IoT a CIMA Hub de forma no invasiva, sin modificar la estructura de datos existente ni introducir dependencias que comprometan la estabilidad actual de la aplicación.

2. Protocolo recomendado: MQTT over WebSockets

Existen dos protocolos principales para este tipo de integración: MQTT (Message Queuing Telemetry Transport) y SNMP (Simple Network Management Protocol). Para este caso de uso se recomienda MQTT por las siguientes razones:

CriterioMQTT ✅ RecomendadoSNMP
Compatibilidad con PWA / browserNativa via WebSocketRequiere puente server
Facilidad de configuraciónBaja complejidadAlta complejidad
Sensores disponibles en el mercadoAmplia variedad (ESP32, Shelly, etc.)Solo equipos de red
Latencia / tiempo realBaja latencia (~100ms)Polling periódico
Costo de hardwareMuy bajo (USD 5–20/sensor)Alto (equipos red)
Integración con FirebaseDirecta via función puenteIndirecta
3. Arquitectura de la solución

La integración se estructura en tres capas independientes. Cada capa puede desarrollarse e incorporarse de forma incremental sin afectar el funcionamiento actual de CIMA Hub:

Sensor IoT→Broker MQTT→Firebase→CIMA Hub
ESP32/Shelly       HiveMQ / Mosquitto    Firestore/RTDB    App existente (sin cambios en lógica)

Capa 1 — Hardware: Sensores IoT

Módulos ESP32 con sensor DHT22 (temperatura + humedad) o BME280 (temperatura + humedad + presión barométrica). Costo estimado: USD 8–15 por sala. Se alimentan con cargador USB. Publican datos cada 30–60 segundos al broker MQTT en formato JSON.

Capa 2 — Broker MQTT

Servidor central que recibe y distribuye los mensajes de los sensores. Opción A (cloud): HiveMQ Cloud, gratuito hasta 100 conexiones, sin infraestructura propia. Opción B (local): Mosquitto en Raspberry Pi o servidor interno, control total sobre los datos. Ambas opciones exponen WebSockets para conectarse desde el browser.

Capa 3 — Integración con CIMA Hub

Una pequeña Cloud Function de Firebase se suscribe al broker MQTT y guarda cada lectura en Firestore bajo una nueva colección ambientales_cima. La app lee esa colección en tiempo real con el listener de Firestore ya existente, sin modificar la estructura de datos de los equipos.

4. Plan de implementación
1
Configurar broker MQTT (HiveMQ Cloud — gratuito)

Crear cuenta en HiveMQ Cloud. Obtener host, puerto WebSocket (8884) y credenciales. Tiempo estimado: 30 minutos.

2
Programar los sensores ESP32

Cargar firmware Arduino que publica cada 60 segundos al topic cima/[sala]/ambiente con payload JSON {"temp":22.5,"hum":58,"pres":1013}. Un ESP32 con BME280 cubre una sala completa.

3
Cloud Function puente MQTT → Firestore

Función Node.js desplegada en Firebase Functions que se conecta al broker, escucha todos los topics cima/# y guarda cada lectura en orgs/cima/ambientales/{sala}/lecturas. Esta función corre en la nube — no requiere servidor propio.

4
Agregar módulo de ambiente a CIMA Hub

Incorporar en la app una nueva sección "Condiciones Ambientales" que lee la colección Firestore en tiempo real. Se muestra temperatura, humedad y presión por sala, con indicadores de color (verde/amarillo/rojo) según rangos aceptables definidos para cada tipo de equipo.

5
Alertas automáticas

Configurar reglas en la Cloud Function para enviar notificación push (Firebase Messaging, ya integrado en la app) si temperatura supera 24°C en sala de RM, humedad baja de 40% en sala de ecografía, etc.

5. Ejemplo — Payload del sensor y estructura Firestore

El sensor publica al topic cima/sala-rm/ambiente el siguiente JSON:

// Payload publicado por el sensor cada 60 segundos
{
  "sala": "sala-rm",
  "temperatura": 21.8,  // °C
  "humedad": 55.2,    // %
  "presion": 1012.4,  // hPa
  "ts": 1749135600   // Unix timestamp
}

// Estructura en Firestore (nueva colección — no toca datos existentes)
orgs / cima / ambientales / sala-rm / lecturas / {autoId}
  → temperatura: 21.8
  → humedad: 55.2
  → presion: 1012.4
  → savedAt: "2026-06-05T14:00:00Z"

8. Impacto en la aplicación existente
✅ Lo que NO cambia
  • Toda la lógica de equipos, mantenimientos, reparaciones y reportes existentes
  • La colección Firestore orgs/cima/app/data (datos actuales) no se modifica
  • Los informes PDF, el inventario y las vistas actuales permanecen intactos
  • La autenticación y los permisos no se ven afectados
🆕 Lo que se agrega
  • Nueva colección Firestore ambientales_cima para lecturas de sensores
  • Una Cloud Function (archivo separado) para el puente MQTT → Firestore
  • Sección "Condiciones Ambientales" en la app (vista nueva, no reemplaza nada)
  • Alertas push opcionales si las condiciones salen de rango
